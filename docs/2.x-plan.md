# senaite.ldap 2.x — autonomy plan

The 1.x line pulls in YAFOWIL purely to render the LDAP control panel
form. The LDAP plugin itself (`pas.plugins.ldap.plugin`) does not need
YAFOWIL — only the `plonecontrolpanel` sub-package does. The 2.x line
replaces the control panel with our own form and drops the YAFOWIL
stack along with its transitive dependencies.

## Where YAFOWIL actually lives in the 1.x chain

```
senaite.ldap.browser.controlpanel.SenaiteLDAPControlPanel
   └─ subclasses pas.plugins.ldap.plonecontrolpanel.controlpanel.LDAPControlPanel
        └─ subclasses pas.plugins.ldap.properties.BasePropertiesForm
             └─ imports yafowil.loader, yafowil.controller.Controller,
                yafowil.yaml.parse_from_YAML, yafowil.base.{ExtractionError,UNSET}
```

`pas.plugins.ldap.plugin` (the actual PAS plugin) has no YAFOWIL import.
That is the only lever we need.

## Goals

1. Drop the entire `yafowil*` family (`yafowil`, `yafowil.plone`,
   `yafowil.yaml`, `yafowil.widget.array`, `yafowil.widget.dict`,
   `webresource`).
2. Drop the runtime dependency on `pas.plugins.ldap.plonecontrolpanel`.
   Keep `pas.plugins.ldap` (the PAS plugin).
3. Keep `pas.plugins.ldap.plugin` and the `node.ext.ldap` stack —
   rewriting either is a separate, much larger discussion.
4. Ship a control panel built from primitives senaite.core already
   uses (`z3c.form` or a plain `BrowserView` + form + `IStatusMessage`).
5. Stay on Python 2.7 / Plone 5.2 / senaite.lims's branch for now.

## Phase 1 — branch & baseline

- Tag current `master` as `1.1.0` (PR #1 + PR #2 already in).
- Cut a `2.x` branch off that tag. `master` continues as the home of
  the 2.x track; `1.x` stays available for backports if a customer is
  pinned.
- Bump `version` in `setup.py` to `2.0.0`.
- Move all `yafowil*` and `webresource` entries to
  `extras_require={"legacy_yafowil": [...]}` so they're installable
  for anyone who still wants the old form during the transition.
- Drop `pas.plugins.ldap.plonecontrolpanel` from
  `setuphandlers.DEPENDENCIES` and from `profiles/default/metadata.xml`
  / `profiles/uninstall/metadata.xml`.

## Phase 2 — own control panel

- Define `ILDAPSettings` in `senaite/ldap/interfaces.py` mirroring the
  fields the YAFOWIL form exposes (`ILDAPProps`: uri, bind dn, bind
  pw, cache, page size, tls, ignore-cert; `ILDAPUsersConfig`: base
  dn, scope, queryFilter, objectClasses, attrmap, rdnAttr, etc.; same
  for groups).
- Replace `senaite/ldap/browser/controlpanel.py` with a Plone-style
  control panel:
  - Either `z3c.form` + `plone.z3cform.layout.wrap_form` (matches how
    senaite.core does its registry panels), or
  - A plain `BrowserView` with a Bootstrap form (matches how some
    other SENAITE panels look).
  - The save handler walks `ILDAPProps(plugin)` /
    `ILDAPUsersConfig(plugin)` / `ILDAPGroupsConfig(plugin)` and
    writes attributes directly — exactly what
    `BasePropertiesForm.save` does today, minus YAFOWIL's extraction
    layer.
- Add a "Test connection" button that calls the same `node.ext.ldap`
  connectivity probe the existing form uses; report OK/error via
  `IStatusMessage`.
- Rewrite `browser/templates/controlpanel.pt`:
  - Drop `<script src="yafowil.plone.js">` and
    `<link href="yafowil.plone.css">`.
  - Use senaite.core's standard prefs layout.
- Drop the `add_bundle_on_request(request, "yafowil")` call.
- Update `setuphandlers.uninstall_pas_plugin`:
  - Remove the `"yafowil.plone"` entry from `DEPENDENCIES`.
  - Re-enable cleanup of any `yafowil` / `plone.bundles/yafowil`
    registry records (currently commented out).

## Phase 3 — dep purge

Update `install_requires` in `setup.py`:

```python
install_requires=[
    "senaite.lims",
    "pas.plugins.ldap==1.8.2",     # plugin only; controlpanel sub-pkg unused
    "node.ext.ldap==1.2",
    "python-ldap==3.3.1",
    "pyasn1==0.4.8",
    "pyasn1-modules==0.2.8",
    "node==1.2",
    "odict==1.9.0",
    "bda.cache==1.3.0",
    "passlib==1.7.4",
    "python-memcached==1.59",
    "plumber<2.0.0",
    "setuptools",
],
```

Six packages gone (`yafowil`, `yafowil.plone`, `yafowil.yaml`,
`yafowil.widget.array`, `yafowil.widget.dict`, `webresource`) plus
their transitive deps.

## Phase 4 — keep the patches in place

The `LDAPSession.search` monkey-patch from PR #2 stays. It only goes
away when a fix lands upstream in `node.ext.ldap`; track the issue,
drop the patch when we can re-pin.

## Phase 5 — migration story

- Upgrade step (GenericSetup profile version bump):
  - Re-import `registry.xml` to drop now-orphan `yafowil` /
    `plone.bundles/yafowil` records.
  - Re-import `cssregistry.xml` / `jsregistry.xml` if they referenced
    YAFOWIL bundles.
  - No data migration on the LDAP side: persistent `pasldap` plugin
    config (`ILDAPProps` / users / groups) stays as-is; only the
    rendering layer changed.
- Uninstall profile stays compatible: still removes the `pasldap` PAS
  plugin and the registry records.
- CHANGES.rst `2.0.0` entry calling out: YAFOWIL removed, new
  control panel UI, no manual action needed on existing pasldap
  config.

## Phase 6 — tests

- Add a layer that installs senaite.ldap + a fake / no-op LDAP plugin
  (no real server).
- Doctest the control panel: GET renders form fields populated from
  `ILDAPProps`, POST persists changes back, status message appears.
- Test that the connectivity probe returns expected shape on success
  and `error` shape on failure (mock
  `LDAPSession.checkServerProperties`).
- Test that `setuphandlers.uninstall_pas_plugin` removes the registry
  records cleanly.

## Risks

- `pas.plugins.ldap.plugin` import side effects — verify by
  inspection that loading `pas.plugins.ldap` without loading
  `pas.plugins.ldap.plonecontrolpanel` doesn't transitively import
  yafowil. (Confirmed for 1.8.2: `plugin.py` is clean; `properties.py`
  is the YAFOWIL importer and is only invoked from the controlpanel
  form, never from the PAS lookup path.)
- pas.plugins.ldap may add a hard yafowil dep in a future release —
  pin to `1.8.2` and don't allow ranges. Watch the upstream changelog.
- Schema drift — `ILDAPProps` etc. are public interfaces of
  pas.plugins.ldap; if upstream changes them, our form needs to follow.
  Same risk as today.

## Out of scope for 2.x

- Replacing `pas.plugins.ldap.plugin` with a SENAITE-owned PAS plugin.
  Means re-implementing user/group enumeration, attribute map, cache
  layer, password authentication. ~1000+ lines of production-critical
  code. Possible but not worth it without a concrete reason.
- Replacing `node.ext.ldap` with a `python-ldap`-only wrapper. Loses
  the schema-aware UGM abstraction. Big rewrite, small payoff.
- Move off Python 2.7 / Plone 5.2. Tied to the senaite.lims roadmap,
  not this package.

## Net result

| Package | 1.x | 2.x |
|---|---|---|
| yafowil | yes | no |
| yafowil.plone | yes | no |
| yafowil.yaml | yes | no |
| yafowil.widget.array | yes | no |
| yafowil.widget.dict | yes | no |
| webresource | yes | no |
| pas.plugins.ldap | yes | yes (plugin only) |
| node.ext.ldap stack | yes | yes |
| python-ldap | yes | yes |

## Sequencing

- PR A — new control panel + drop pas.plugins.ldap.plonecontrolpanel
  from ZCML, with a feature flag (env var
  `SENAITE_LDAP_USE_LEGACY_CONTROLPANEL=1`) so we can fall back during
  testing.
- PR B — purge YAFOWIL from `install_requires`, drop the registry
  records, finalise the upgrade step.
